## Detailed Application of Each Principle

### Principle 1.2: Avoid Harm

**Potential harm if this were deployed uncritically:**

A sports betting company might use a sentiment model to set odds. If they only reported the significant xG finding (β = 0.448, p = 0.044) without also reporting that sentiment **does not predict wins** (p = 0.420), bettors could be misled into overvaluing sentiment.

**How this audit prevents harm:**

- Null findings are presented as prominently as significant findings.
- The README explicitly states: "Sentiment alone is insufficient for betting decisions."
- Model diagnostics are fully disclosed, showing the model's limitations.

**Harm that cannot be fully prevented:**

A bad actor could ignore this ethics statement and use the code irresponsibly. This is true of any open-source project. The mitigation is clear licensing (MIT with an ethical use recommendation) and public documentation that any responsible deployer would cite.

---

### Principle 1.3: Be Honest and Trustworthy

**Honest reporting checklist (all checked):**

- ✅ All p-values reported (no cherry-picking)
- ✅ Confidence intervals included (95% CI for sentiment: [0.012, 0.884])
- ✅ Non-significant results disclosed (logistic regression sentiment p = 0.420)
- ✅ Model assumptions tested and reported (normality, homoscedasticity, independence, multicollinearity)
- ✅ Data limitations disclosed (single team, single season, Twitter/X bias)

**What a dishonest version would look like (for contrast):**

A dishonest analyst might only publish the linear regression (sentiment significant) and omit the logistic regression (sentiment not significant). This audit explicitly refuses that practice.

**Trustworthiness in practice:**

All code is provided. Any reader can reproduce the results. No "black box" models (neural networks, proprietary APIs) are used — only transparent regression models with interpretable coefficients.

---

### Principle 1.6: Respect Privacy

**Data provenance:**

| Data type | Source | Privacy status |
|-----------|--------|----------------|
| Performance data | FBref.com (publicly available match statistics) | No personal data |
| Sentiment data | Twitter/X API (public tweets) | Public data only; no DMs; no private accounts |

**Specific privacy measures taken:**

1. **No user identification:** Tweets were aggregated to match-level averages. No individual usernames, user IDs, or tweet text are stored in the final dataset.
2. **No re-identification possible:** The released data contains only `Avg_Sentiment` per match — not per tweet or per user.
3. **Rate limiting compliance:** API calls respected Twitter/X rate limits, avoiding excessive server load.

**What would violate privacy (not done):**

- Collecting tweets from users who had "protected" accounts
- Storing raw tweet text with user handles in a public repository
- Using sentiment to infer individual psychological states

**Future privacy consideration:**

If this analysis were expanded to include player-specific sentiment (e.g., "@MarcusRashford"), additional privacy review would be required. Player sentiment could affect mental health. That expansion is explicitly **not** part of this audit.

**Applicable laws considered:**

|--------------|-----|-------------------|

**Best practice adopted (beyond legal minimum):**

Even where not legally required, this project:
- Does not retain raw tweet text
- Does not sell or share data with third parties
- Provides this ethics statement so downstream users understand their obligations
